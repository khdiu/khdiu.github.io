3种缓存读写策略

Cache Aside Pattern（旁路缓存模式）

也称之为Lazy-Loading模式

Cache Aside Pattern是平时使用较多的一个缓存读写模式，比较适合请求比较多的场景

Cache Aside Pattern中服务端需要同时维系DB和cache，并且是以DB的结果为准

在Cache Aside模式中，数据的读写都是由应用程序主动控制，而缓存并不自动维护数据的一致性。以下是Cache Aside模式的基本工作原理和要点

缓存读写步骤：


写：

  1. 先更新DB
  2. 然后直接删除Cache

读：

  1. 从Cache中读，命中就直接返回
  2. 未命中就从DB中读取数据返回
  3. 把数据更新到Cache中

本模式是先更新DB，后删除Cache。

无论是先更新DB还是先删除Cache都可能造成缓存不一致问题。


Cache Aside模式的优点在于简单易于实现，适用于许多不同类型的存储后端，以对数据库访问的灵活控制。

该模式存在以下缺点

  • 缓存一致性问题：Cache Aside模式不自动维护数据一致性，需要应用程序开发者确保数据的有效性和一致性
  • 缓存更新延迟：如果数据在主存储中被修改后，需要等待缓存过期或明确的刷新操作才能获取最新数据，可能导致数据不一致的问题
  • 缓存过期管理：需要定期清理和维护缓存，以确保缓存中的数据不过期或占用太多内存

Cache Aside模式通常适用于对数据一致性要求较低的场景，例如读取操作频繁，但写入操作较少的应用程序，在需要更强一致性和实时性的情况下，需要使用其他缓存模式


缓存一致性维护

先更新DB还是先删除缓存？

缓存一致性维护的重点在于减少缓存与主存储数据不一致的时间，缓存不一致无法避免，只能减少不一致的时间

常见误区：维护缓存一致性的解决的主要问题并不是增加用户访问新数据的访问速度，而是减少缓存与主存储数据不一致的时间，切勿混淆选择原因


方案1：先删除cache，后更新DB

此种方式先删除cache，当DB更新还未完成的时候，如果有请求到达，则会请求DB中的旧数据，然后DB会返回旧数据并写入cache，DB更新完成后出现数据不一致问题，数据不一致的时间为一个TTL（其他开销时间不变）。

缓存不一致的场景：

请求1先把cache中的A数据删除 -> 请求2从DB中读取数据A -> 请求1再把DB中的A数据更新

由于磁盘操作的速度远小于缓存操作的速度，在进行DB写入的同时可能会出现新的DB读取，此时磁盘更新没有完毕，请求2获取到了DB中过时的数据，出现缓存不一致


方案2：先更新DB，后删除cache

此种方式先更新DB，后删除cache，在其他在DB更新完成后，在进行cache更新，此种方式缓存不一致的时间为缓存数据的写入时间。

由于缓存读写较快，此种方式下缓存的不一致时间通常远远小于方案1导致的一个TTL时间的缓存不一致问题

缓存不一致的场景：

请求1从DB读数据A -> 请求2写更新数据A到数据库并删除cache中的A数据 -> 请求1将数据A写入cache


Cache Aside Pattern的缺陷

缺陷1：首次请求数据一定不存在cache的问题

解决方法：将热点数提前放入cache中

缺陷2：写操作比较频繁的话会导致数据被频繁删除，这样会影响缓存命中率

解决方法：

  • 数据库和缓存数据强一致场景：更新DB的时候同时更新cache，使用锁/分布式锁来保证二者一起更新的原子性
  • 短暂允许数据库和缓存数据不一致的场景，更新DB的时候同时更新cache，不需要加锁保证原子性，但是给缓存加一个比较短的过期时间，这样可以保证数据即使不一致的话影响也比较小

Read/Write Through Pattern（读写穿透）

Read/Write Through Pattern中服务端把cache视为主要数据存储，从中读取数据并将数据写入其中。cache服务负责将此数据读取和写入DB，从而减轻了应用程序的职责


写（Write Through）：

  • 先查cache，cache中不存在，直接更新DB
  • cache中存在，则先更新cache，然后cache服务自己更新DB（同步更新cache和DB）

读（Read Through）

  • 从cache中读取数据，读取到就直接返回
  • 读取不到的话，先从DB加载，写入到cache后返回响应

Read-Through Pattern 实际上只是在Cache-AsidePatter上进行了封装。

在Cache-Aside Pattern下，发生请求的时候，如果cache中不存在对应的数据，是由客户端自己负责把数据写入cache。

而Read Through Pattern则是在cache服务自己来写入缓存，这对客户端是透明的。


Read-Through Pattern也有首次请求数据一定不在Cache的问题，对于热点数据可以提前放入缓存中


Write Behind Pattern（异步缓存写入）

Write Behind Pattern 和 Read/Write Through相似，两者都是由cache服务来负责cache和DB的读写


同时，二者又存在很大的不同：

Read/Write Through 是同步更新cache和DB。

Write Behind Caching则是只更新缓存，不直接更新DB，而是改为异步批量的方式来更新DB。


明显，这种方式对数据一致性带来了更大的挑战，比如cache数据可能还没异步更新DB的话，cache服务可能就挂掉了。


这种策略在平时开发过程中比较少见，但是不代表应用场景少，比如消息队列中消息的异步写入磁盘、MySQL的InnoDB Buffer Pool机制都使用到了这种策略。


Write Behind Pattern下DB的写性能非常高，非常适合一些数据经常变化，又对数据一致性要求不高的场景，比如浏览量、点赞量


该模式的优点包括了应用程序的响应性、降低了写入延迟、并能够高校处理大量的写入操作。但是，也存在以下挑战：

  • 数据一致性：Write-Behind引入了潜在不一致性的时段，既缓存与主数据存储之间的数据可能不同步。如果应用程序崩溃或写后处理失败，数据可能不会同步
  • 增加复杂性：实现后台写入的逻辑可能较为负责，需要处理写入失败，异步处理、数据一致性等问题
  • 写操作批处理的管理：管理批处理操作可能需要考虑事务性和错误处理
